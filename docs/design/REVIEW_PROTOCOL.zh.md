# 评审协议

> 每个 PR 的强制质量关卡。面向贡献者的摘要见[贡献指南](../wiki/Contributing.md);
> 本文是完整协议及其理念。

---

## 0. 为什么

工具包靠不断加入基准、指标、评审而成长——每一次都可能悄悄破坏论文对齐、双入口
逐位一致契约,或注册表不变量。统一的关卡让正确性不依赖某位评审者的记忆。

## 1. 三层

| 层 | 关卡 |
|---|---|
| **1. CI** | ruff · pytest · doc-links · 设计文档一致性 · PR 模板检查 |
| **2. 自查** | PR 模板里的 12 项清单,全部勾选或标 N/A |
| **3. 同行评审** | 一位评审者执行 5 问检查 |

三关全过,PR 才能合并。**在合并命令*之前*确认完整回归全绿,而非之后**——
hook 失败意味着提交并未发生。

## 2. PR 类型标签(至少选 1)

每个标签在 CI 之外追加一道验收关:

| 标签 | 额外关卡 |
|---|---|
| `new-metric` | 论文对齐 ± 容差 · 标签在词表内 · `metric show` 字段 |
| `lift-out` | 相对基准路径**逐位一致 ≤ 1e-6** · 基准回归 |
| `new-bench` | 冒烟 + 集成 · `dim_tags` · TEST_MANUAL 行 |
| `new-judge` | `judges test` · ≥ 1 次基准运行 · 定价 |
| `schema-change` | 迁移测试 · 版本号提升 |
| `cli-change` | 更新 `--help` · USER_MANUAL 示例 |
| `doc-only` / `infra` / `refactor` | 按本协议适用更轻的关卡 |

## 3. 5 问同行检查

1. 哪份设计文档 / wiki 页面规划了此改动?
2. 该类型标签的验收关卡通过了吗?
3. 它会破坏哪条现有路径?
4. CI 是绿的吗?
5. 30 天后该决策是否可追溯(文档 / memory)?

## 4. 周期一致性

除逐 PR 评审外,`check_design_doc_consistency.py` 守护跨切面不变量
(能力分类法大小 = 44、指标标签在词表内、注册表必填字段、每个用评审的基准都声明
`paper_judge`)。这能抓到任何单个 PR 评审都看不出的漂移。

## 5. 维护 agent 承诺

长期维护 agent 遵循:intake 报告 → 用户批准 → 逐位一致 + 论文对齐的集成 →
每项一个 PR,每个都过此关卡。该行为已 skill 化于
`.claude/skills/videvalkit-maintainer/`。
