# 快速评测与训练监控 — 设计

> 快速、已校准评测的理念。用法见
> [配置档与快速评测](../wiki/guides/Profiles-and-Quick-Eval.md)与
> [训练监控](../wiki/guides/Training-Monitor.md)。

---

## 1. 问题

论文忠实运行要数小时——远不够快,无法每 N 个训练步调用。但快速评测只有在其排序
与全量评测*一致*时才有用;未校准的子集会误导。所以设计需要两样东西:一个速度旋钮,
以及该旋钮保留排序的证据。

## 2. 目标 / 非目标

- **目标**:一个分钟级、能追踪全量趋势的配置档;一种证明这种一致性的方法;一个
  不 shell 外调、不阻塞的训练循环 API。
- **非目标**:替代论文忠实评测,或发明新指标。配置档与*评测什么*(基准/指标)、
  *用哪个评审*正交。

## 3. 评测配置档

精度↔速度曲线上的三个具名点,控制 prompt 子集、帧采样、每 prompt 样本数:

| 配置档 | 子集 | 帧数 | 样本数 | 目标 |
|---|---|---|---|---|
| `quick` | 小 | 4 | 1 | 训练监控 / 冒烟 / CI;相对全量 ρ ≥ 0.85 |
| `standard` | 中 | 8 | 1 | 消融 / 迭代;ρ ≥ 0.95 |
| `full` | 全 | 8 | 5 | 论文 / 榜单(默认) |

不带 `--profile` = `full`,故既有行为不变。

## 4. 子集校准

子集是一份**版本锁定的** `prompt_id` JSON,外加溯源(SHA-256)与校准记录。方法:

1. 拉取全语料榜单分数。
2. 提出候选子集(按维度分层)。
3. 跨模型计算子集排序与全量排序之间的 Spearman ρ。
4. 仅当 ρ ≥ 配置档阈值(quick 为 0.85)时固化子集。

锁定 + 溯源意味着 `quick` 数字可复现,且其与 `full` 的一致性是记录在案的事实,
而非期望。`--subset <file>` 覆盖配置档的默认子集。

## 5. 规划与多基准

- `estimate` 在运行*之前*预览墙钟 / GPU 小时 / 评审调用数(设了评审时还有
  token/$)——成本一目了然。
- `eval-suite` 把多个基准(或 `--all-anchored`)跑进一个 workspace,每个按需用
  各自的配置档/评审。

## 6. 训练循环集成

`videvalkit.training.monitor` 是显式 API,而非框架回调:

- `monitor.preview_prompts(cfg)` → 所配置基准+配置档需要的 prompt 集
  (精确生成这些)。
- `monitor.eval(videos, model_name, cfg, step)` → 跑基准/指标,向
  `timeline.jsonl` 追加一行,返回 `MonitorResult`。

`MonitorConfig` 带 `metrics=` 字段,使独立指标层直接接入。CLI `watch` 轮询
检查点 glob 并评测每个新检查点。跨基准 `overall` 是各基准 overall 的原始均值
(趋势信号;z-score 归一化是后续改进)。监控中的 FVD 默认用 S3D 骨干——
无需论文 I3D 权重的有效趋势指标。
